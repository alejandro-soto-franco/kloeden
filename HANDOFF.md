# kloeden — handoff

## What this is

A C++ reference implementation and benchmark companion for two existing Rust crates:

- **elworthy** — Cranelift JIT for Bismut-Elworthy-Li Monte Carlo Greeks on non-stationary SDEs. Repo: https://github.com/alejandro-soto-franco/elworthy. Local: `~/elworthy`.
- **pathwise** — parallel Rust SDE simulation toolkit (Euler-Maruyama, Milstein, Taylor 1.5 / SRI). Repo: https://github.com/alejandro-soto-franco/pathwise. Local: `~/pathwise`.

Naming: Peter Kloeden, co-author of Kloeden-Platen, *Numerical Solution of Stochastic Differential Equations* (Springer 1992), the canonical reference for the schemes implemented in pathwise.

## Why it exists (the one-sentence thesis for the resume / blog)

> "Hand-written SIMD C++ matched against Rust with LLVM and Cranelift backends on the same MC-Greeks + SDE workloads; Rust is within noise of C++ and, for non-smooth payoffs, correct where naive C++ pathwise differentiation is biased."

Deliverables that carry that thesis:
1. A C++ port of representative elworthy + pathwise kernels
2. A rigorous benchmark harness (single-thread throughput, SIMD width sweep, cold vs. warm kernel)
3. A short writeup suitable for sotofranco.dev and as a talking point for C++-heavy quant interviews (QRT, G-Research, XTX, Optiver)

## Success criteria (in priority order)

1. **Correctness parity.** C++ and Rust implementations produce bitwise-close (or statistically indistinguishable, given RNG) outputs on a shared fixture set. Test harness lives in the repo.
2. **Headline benchmark number.** One credible, reproducible table: rows = scheme (Euler-Maruyama, Milstein, Taylor 1.5, BEL-Delta on digital), columns = implementation (Rust-LLVM, Rust-Cranelift-JIT, C++ scalar, C++ SIMD). Wall-clock ns/path and paths/s, single-thread, one pinned core.
3. **Non-smooth-payoff demonstration.** At least one case (digital or barrier option) where naive C++ pathwise differentiation returns zero/biased estimate while elworthy's likelihood-ratio Malliavin weight returns an unbiased one. This is the "correctness matters, not just speed" story.
4. **Blog-ready writeup.** ~2000-word post draft with methodology, numbers, and one figure. Target publication to sotofranco.dev (sotofranco.dev repo is at `~/sotofranco.dev`; follow the existing MDX conventions there).

## Scope — what to build

### C++ side (`cpp/`)

- CMake-based build, C++20 (concepts + `<numbers>` useful), `-O3 -march=native`, `-ffast-math` *only* in a clearly-labelled fastmath build target (fastmath changes numerical behaviour; keep a strict-IEEE target too).
- Scalar + SIMD (AVX2 and AVX-512 where available; `std::experimental::simd` if available, else intrinsics or xsimd).
- Reference implementations:
  - Euler-Maruyama, Milstein, Taylor 1.5 for GBM, Heston, OU (match pathwise's built-in process list).
  - Bismut-Elworthy-Li Monte Carlo Delta for a digital payoff on GBM. This is the headline non-smoothness case.
- Link options: (a) pure standalone C++ for baseline, (b) `cxx`-bridged C++ calling into Rust and vice versa for FFI-overhead measurement. The `cxx` side doubles as the "how Rust slots into legacy C++" demo.
- Depend on as few third-party libraries as possible. `xsimd` is acceptable if portable SIMD needs it. Prefer `std::random` over `boost::random` unless the RNG quality comparison matters (it probably does for MC; document the choice).

### Benchmark harness (`bench/`)

- Google Benchmark for C++, Criterion for Rust. Both already familiar from elworthy / pathwise.
- Pinned-core wrapper script (`taskset -c 2` on Fedora) to reduce variance.
- Fixed RNG seeds; fixed path counts; report median + MAD not mean + stddev.
- Output format: a CSV checked into the repo under `results/` with a `README.md` describing the host CPU, governor, turbo state, kernel, compiler versions.

### Rust side (`rust/`)

- Minimal wrapper Cargo workspace that pulls `elworthy` and `pathwise` from crates.io (pinned versions) and exposes the same kernels in a form directly comparable to the C++ side.
- Include both LLVM-backend and Cranelift-JIT numbers for elworthy; only LLVM for pathwise (pathwise isn't JIT-based).

### Writeup (`docs/`)

- `methodology.md`: how the benchmark is run, what was controlled for, what caveats apply (compiler version skew, fastmath effects, RNG choice, path count for MC variance).
- `results.md`: the headline table + one speed/accuracy scatter for the digital-delta case.
- `blog-draft.md`: the 2000-word sotofranco.dev post, matching blog conventions (see `~/sotofranco.dev/`).

## Non-goals

- Not a production C++ Monte Carlo library. No Boost, no QuantLib dependency, no serialization, no option-pricing framework beyond what's needed for the benchmark.
- Not a Rust→C++ transpiler or autogenerated port. Every C++ kernel is hand-written against the mathematical spec, not mechanically translated from Rust.
- Not a multi-threaded benchmark (yet). Single-thread, pinned core is the story; rayon parallelism in pathwise is a known multiplier and doesn't need re-demonstration.
